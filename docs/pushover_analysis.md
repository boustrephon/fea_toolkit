# Pushover (Non‑linear Static) Analysis

## Overview

Pushover analysis is a non‑linear static procedure used to evaluate the
seismic performance of structures.  A monotonically increasing lateral load
(or displacement) is applied to the structure while gravity loads are held
constant.  The result is a **capacity curve** — base shear vs. control node
displacement — that shows how the structure stiffens, yields, and degrades
under increasing lateral demand.

The fea_toolkit implementation follows the standard OpenSees two‑stage
approach used in well‑known examples (e.g. the
[PEER RC Frame pushover](https://github.com/OpenSees/OpenSees/blob/master/EXAMPLES/Example3/Example3.1%20-%20Pushover.tcl)
and the included
`docs/references/RCFramePushOver_v2.py`):

```
Stage 1:  Apply gravity loads (load‑controlled)
Stage 2:  Lock gravity (loadConst), apply lateral loads, push
          (displacement‑controlled)
```

---

## How it works

### Stage 1 — Gravity

1. The model is (re-)built with the configured element type and sections.
   If all frame sections support fiber patches (`to_fiber_patches()` returns
   patches rather than raising `NotImplementedError`), the builder
   automatically uses:

   - `forceBeamColumn` elements with `Lobatto` beam integration
   - **Fiber sections** (`Steel01` / `Concrete01` uniaxial materials)
   - This enables non‑linear material behaviour (yielding, plastic hinge
     formation)

   Otherwise it falls back to `elasticBeamColumn` with elastic sections
   (linear pushover).

2. All load patterns listed in `gravity_patterns` are created and their
   loads applied (joint loads, frame self‑weight, area self‑weight,
   area gravity loads, frame gravity loads, frame distributed loads).

3. A load‑controlled static analysis is run using
   `LoadControl(1.0)`:

   ```python
   ops.integrator('LoadControl', 1.0)
   ops.analysis('Static')
   ops.analyze(1)
   ```

   The structure deforms under gravity alone.

### Stage 2 — Lateral push

4. **`ops.loadConst('-time', 0.0)`** is called.  This is the critical
   command that separates the two stages:

   - All *existing* load patterns (gravity) are **locked** at their
     current load factor (1.0).  They will remain active at that level
     for all subsequent analysis steps.
   - The domain **time** is reset to 0.0.  Any new patterns created
     after this point start from time 0.0 and are free to vary.

   Without `loadConst`, a new lateral pattern would be mixed with the
   gravity patterns and scaled uniformly by the integrator, producing
   incorrect results (e.g. gravity loads appearing as lateral reaction).

5. A new **lateral reference pattern** is created.  The **shape** of the
   lateral load is determined by the `lateral_load_type` parameter:

   - **`'uniform'`** — nodal loads proportional to mass
     ($F_i \propto m_i$) generated by `_compute_uniform_lateral_loads()`.
   - **`'triangular'`** — nodal loads proportional to mass × height$^k$
     ($F_i \propto m_i h_i^k$) generated by `_compute_triangular_lateral_loads()`.
   - **`'mode1'`** — nodal loads proportional to mass × mode shape
     ($F_i \propto m_i \phi_i$) generated by `_compute_mode_shape_lateral_loads()`.
   - **`'pattern'`** — distributed loads read from an existing SAP2000
     load pattern (the original approach).

   For the first three types, loads are applied as **concentrated nodal
   loads** (`ops.load`).  A dedicated time series and pattern tag are
   created for the lateral load (no tag collision with gravity patterns).

6. The integrator is switched to **`DisplacementControl`**:

   ```python
   ops.integrator('DisplacementControl', control_node, dof, disp_inc)
   ops.analysis('Static')
   ```

   This drives the control node in the specified DOF by `disp_inc` at
   each step, finding the load factor on the lateral pattern that
   produces the target displacement.

7. The push proceeds in a loop:

   ```python
   for step in range(num_steps):
       ok = ops.analyze(1)
       # Record control node displacement and base shear
   ```

   At each step the control displacement and base shear (sum of reactions
   at restrained nodes in the push direction) are recorded, producing the
   **capacity curve**.

## Key commands explained

| Command | Purpose |
|---|---|
| `ops.loadConst('-time', 0.0)` | Lock gravity loads; reset time so new patterns start from zero |
| `ops.integrator('LoadControl', 1.0)` | Ramp all loads to full value in one step (gravity stage) |
| `ops.integrator('DisplacementControl', node, dof, inc)` | Drive a node's displacement; scale lateral load to achieve it |
| `ops.timeSeries('Linear', tag)`, `ops.pattern('Plain', tag, tag)` | Create a dedicated time series + lateral load pattern (separate from gravity pattern tags) |
| `ops.load(tag, fx, fy, fz, ...)` | Concentrated nodal load (used for `uniform`, `triangular`, `mode1`) |
| `ops.eleLoad('-ele', tag, '-type', '-beamUniform', ...)` | Element‑span load (used for `pattern` type) |
| `Steel01(matTag, Fy, E, b)` | Steel material with kinematic hardening (used for fiber sections) |

## Lateral load types — what they are and why to use them

Both **ASCE 41-17 §7.2** and **ATC-40 §2.2** require at least **two**
distinct lateral load patterns to bracket the possible inertial force
distribution as the structure yields and degrades.  The four options in
`run_pushover_analysis()` are:

| # | Type | Shape | Requires | Code status |
|---|---|---|---|---|
| 1 | `'uniform'` | $F_i \propto m_i$ | Seismic masses | ✅ ASCE 41 / ATC‑40 required |
| 2 | `'triangular'` | $F_i \propto m_i h_i^k$ | Masses + optional $T_1$ | ✅ ASCE 41 / ATC‑40 required |
| 3 | `'mode1'` | $F_i \propto m_i \phi_{i1}$ | Masses + mode shapes | ✅ Allowed alternative |
| 4 | `'pattern'` | From SAP2000 `eleLoad` | A named load pattern | ❌ Not code‑compliant |

### 1. Uniform (mass‑proportional) — `'uniform'`

**Shape:** $F_i \propto m_i$

Every node with mass receives a lateral force proportional to its mass.
This is equivalent to applying a **uniform ground acceleration** to the
structure — every mass particle feels the same acceleration.

**When to use:** Required by both ASCE 41 and ATC-40 as one of the two
minimum patterns.  The uniform pattern places **more demand on lower
storeys** compared to the triangular pattern, because upper-storey masses
(the smallest contributor to the triangular pattern's $m_i h_i^k$ factor
at small $h$) still receive full lateral force.  It often produces larger
base shear and can reveal strength or ductility weaknesses at lower levels
that a triangular pattern might underestimate.

**Typical result:** Higher base shear, lower roof displacement compared
to triangular for the same structure.

### 2. Triangular (ELF / code pattern) — `'triangular''

**Shape:** $F_i \propto m_i h_i^k$

Lateral force increases with both mass and **height above the base**,
following the same distribution as the ASCE 7 Equivalent Lateral Force
(ELF) procedure.  The exponent $k$ accounts for higher-mode effects in
taller structures:

- $k = 1.0$ for $T \le 0.5$ s (low‑rise, stiff structures)
- $k = 2.0$ for $T \ge 2.5$ s (tall, flexible structures)
- Linear interpolation for $0.5 < T < 2.5$ s

**When to use:** Required as the second minimum pattern in both ASCE 41
and ATC-40.  The triangular pattern approximates the fundamental‑mode
inertial force distribution and tends to produce **larger roof
displacements** than the uniform pattern, making it more demanding on
upper‑storey drift-sensitive components.

**Typical result:** Lower base shear, larger roof displacement compared
to uniform.

### 3. Mode‑shape pattern — `'mode1'`

**Shape:** $F_i \propto m_i \phi_{i1}$

Lateral force is proportional to the product of nodal mass and the
**fundamental mode eigenvector** component in the push direction.  This
is the most accurate representation of the elastic inertial force
distribution.

**When to use:** Allowed as an alternative to the triangular pattern in
ASCE 41.  Particularly useful when:
- The fundamental mode shape is not well‑approximated by a triangle
  (e.g., structures with set‑backs, transfer storeys, or non‑uniform
  mass distribution).
- You want to compare results against a response‑spectrum analysis
  that uses the same mode shape.

**Requires:** A prior call to `run_modal_analysis()` followed by
`extract_mode_shapes()`.

### 4. Pattern-based (SAP2000) — `'pattern'`

**Shape:** Taken directly from an existing SAP2000 load pattern's frame
distributed loads (`eleLoad` commands).

**When to use:** Kept for backward compatibility and for specialised
cases where the user wants complete control over the load distribution
(e.g., a custom user‑defined pattern, or a pattern exported from another
source).  Not recommended for code‑compliant seismic assessment.

### Comparing the patterns

```
Lateral force distribution (qualitative)
                                              ▲
                    uniform                   │
                 mass-proportional            │
                                              │   ●  ●  ●  ●  ●  ●
                                              │
                    triangular                │    height
                 (ELF, k=1..2)                │      ●
                                              │        ●
                                              │          ●
                                              │            ●
                                              │              ●
                                              └──────────────────► force
```

**Recommendation:** Run at least two pushover analyses — one `'uniform'`
and one `'triangular'` — and use the envelope of both capacity curves for
performance assessment, as required by ASCE 41-17.

## Using the `k` exponent (triangular pattern)

The $k$ exponent in the triangular pattern follows ASCE 7-16 §12.8.3:

```
k = 1.0                     if  T₁ ≤ 0.5 s
k = 2.0                     if  T₁ ≥ 2.5 s
k = 1.0 + (T₁ - 0.5) / 2.0  if  0.5 < T₁ < 2.5 s
```

Where $T_1$ is the fundamental period from modal analysis.  If
`fundamental_period` is not provided (left as `None`), $k = 1.0$ is used
(a conservative choice for most low‑ to mid‑rise structures).

## Usage

### Basic — mass‑proportional (uniform) pattern

```python
from fea_toolkit.opensees.builder import OpenSeesBuilder
from fea_toolkit.plotting import plot_pushover_curve

builder = OpenSeesBuilder(model_data, config)
builder.build()
builder.compute_seismic_masses()
modal = builder.run_modal_analysis(num_modes=3)

results = builder.run_pushover_analysis(
    gravity_patterns={'DEAD': 1.0, 'DEAD SDL': 1.0, 'LL': 1.0},
    lateral_load_type='uniform',
    lateral_direction='X',
    max_disp=0.5,
    num_steps=50,
)

fig = plot_pushover_curve(results)
fig.savefig('pushover_curve.png')
```

### Triangular (ELF) pattern

Uses the ASCE 7 equivalent lateral force distribution $F_i \propto m_i h_i^k$:

```python
results = builder.run_pushover_analysis(
    gravity_patterns={'DEAD': 1.0, 'DEAD SDL': 1.0, 'LL': 1.0},
    lateral_load_type='triangular',
    lateral_direction='X',
    fundamental_period=modal['periods'][0],  # T₁ from modal analysis
    max_disp=0.5,
    num_steps=50,
)
```

### Mode‑shape pattern

Uses the fundamental mode eigenvector: $F_i \propto m_i \phi_{i1}$:

```python
shapes = builder.extract_mode_shapes(modal['num_modes'])

results = builder.run_pushover_analysis(
    gravity_patterns={'DEAD': 1.0, 'DEAD SDL': 1.0, 'LL': 1.0},
    lateral_load_type='mode1',
    lateral_direction='X',
    mode_shapes=shapes,
    mode_index=0,
    max_disp=0.5,
    num_steps=50,
)
```

### Pattern‑based (SAP2000 load pattern)

Uses an existing SAP2000 load pattern's distributed loads as the lateral
load shape (kept for backward compatibility):

```python
results = builder.run_pushover_analysis(
    gravity_patterns={'DEAD': 1.0, 'DEAD SDL': 1.0, 'LL': 1.0},
    lateral_load_type='pattern',
    lateral_pattern_name='Wind +X',
    lateral_direction='X',
    max_disp=0.5,
    num_steps=50,
)
```

### With fiber sections (non‑linear)

If your model's frame sections support `to_fiber_patches()` (ISection,
PipeSection, BoxSection, RectangularSection, CircularSection), the builder
automatically uses `forceBeamColumn` with fiber sections.  No additional
configuration is needed — the fallback to elastic is automatic.

### Controlling the lateral load shape

For ``'uniform'``, ``'triangular'``, and ``'mode1'`` types, the lateral
loads are generated from the model's mass distribution and applied as
**concentrated nodal loads** (`ops.load`).  ``DisplacementControl`` then
scales the entire pattern uniformly to achieve the target displacement.
For ``'pattern'`` type, the loads come from the specified SAP2000 load
pattern's frame distributed loads.

## Output

`run_pushover_analysis()` returns a dictionary with:

| Key | Description |
|---|---|
| `step` | Step numbers (0 = after gravity) |
| `control_disp` | Control node displacement relative to gravity state |
| `base_shear` | Total base shear in the push direction at each step |
| `status` | Analysis return code (0 = success) |
| `gravity_displacements` | Nodal displacements after gravity only |
| `control_node` | The control node tag used |
| `dof` | The push DOF (1=X, 2=Y, 3=Z) |
| `lateral_load_type` | The load pattern type used (`'uniform'`, `'triangular'`, `'mode1'`, or `'pattern'`) |
